# TK-OPS V2 前端需求文档

## 1. 文档定位

本文用于把 V2 产品问题拆成前端可执行需求。  
文档不写实现代码方案，但要写到 implementer 无需再猜页面应如何表现。

## 2. 跨页面前端 V2 要求

### 2.1 页面首屏信息优先级

所有核心页面首屏必须优先回答四个问题：

1. 我现在在处理哪个项目或对象？
2. 当前页面最重要的可继续动作是什么？
3. 当前任务或数据状态是什么？
4. 如果不能继续，缺什么以及去哪里补？

### 2.2 阻断态到引导态的统一规则

- 禁止只显示“未配置”“不可用”“请先完成前置步骤”。
- 必须同时给出原因、影响范围、下一步和可执行按钮。
- 引导按钮命名必须明确，例如“去配置 Provider”“去选择项目”“重新检测 Runtime”。

### 2.3 保存 / 执行 / 失败 / 重试统一反馈

- 保存成功必须可见，且显示保存时间或版本变化。
- 执行开始必须可见，且能看到处理中状态。
- 失败必须可见，且给出失败原因与重试入口。
- 可取消操作必须提供取消反馈，不允许无感知中断。

### 2.4 共性状态呈现规范

- 未选择项目：统一使用项目上下文缺失引导。
- Provider 未配置：统一展示受影响能力、去配置入口、测试入口。
- 任务处理中：统一展示任务归属、阶段、最近更新时间。
- 空状态：统一说明当前没有数据和下一步动作。
- Runtime 错误：统一展示中文提示、错误来源和重试按钮。

### 2.5 文案与控件规则

- 按钮命名统一使用动作动词，不用模糊词。
- 状态标签统一使用“未开始 / 处理中 / 已完成 / 失败 / 可重试 / 已取消”。
- 空态说明统一采用“当前没有什么 + 你现在可以做什么”的句式。

### 2.6 Light / Dark 双主题要求

- 同一状态在双主题下语义一致。
- 禁止只在单主题下可读或可区分。
- 核心状态色、边界、禁用态和高亮态都必须同时验收。

## 3. 逐页前端需求

### 3.1 首启与许可证向导

- 用户目标：完成许可、目录初始化与 Runtime 健康检查。
- 当前体验问题：失败时被阻断，恢复路径不清。
- 页面信息结构改动：把校验项拆成步骤列表，首屏展示当前步骤与剩余步骤。
- 组件 / 交互要求：每个失败项提供独立重试或修复入口。
- 状态与反馈要求：明确展示检测中、检测成功、检测失败、重新检测中。
- 依赖的 Runtime 能力：许可校验、目录初始化、Runtime 诊断。
- 页面验收标准：用户能独立完成首启流程或明确修复路径。

### 3.2 创作总览

- 用户目标：快速回到当前项目与最近任务。
- 当前体验问题：信息多，但继续动作不突出。
- 页面信息结构改动：首屏固定”继续创作””最近任务””待处理风险”。
- 组件 / 交互要求：项目卡、任务卡、快速跳转入口统一样式。
- 状态与反馈要求：无项目时引导创建或进入项目；有失败任务时显著提示。
- 依赖的 Runtime 能力：项目摘要、任务摘要、最近活动。
- 页面验收标准：用户能从总览直达当前工作。

#### 3.2.1 实测缺陷（前端侧）

- **Bug F-01 项目删除未落库**：`CreatorDashboardPage.vue:271-284` 的 `handleDeleteProject` 只在前端过滤 `projectStore.recentProjects`，没有调用 Runtime 删除接口；注释 `Fallback deletion: filter out locally since runtime API for deletion isn't strictly defined for dashboard summary` 直接说明绕过了后端。现象：用户点删除后重启桌面端项目又回来。
- 修复方向：接入后端即将补齐的 `DELETE /api/dashboard/projects/{id}`（详见后端文档 Bug B-01），前端改为 `await projectStore.deleteProject(projectId)`，成功后再从列表移除；失败必须弹中文错误并恢复列表。
- 前端验收：删除项目后重启桌面端该项目不再出现；删除失败时列表未被错误移除且有可重试提示。

### 3.3 脚本与选题中心

- 用户目标：生成、编辑、保存、改写脚本。
- 当前体验问题：版本与保存结果不透明。
- 页面信息结构改动：首屏分为当前脚本、版本状态、AI 动作区。
- 组件 / 交互要求：保存状态条、版本切换、改写动作、草稿提示统一化。
- 状态与反馈要求：生成中、保存中、已保存、失败可重试必须显式。
- 依赖的 Runtime 能力：脚本内容、版本信息、生成任务状态。
- 页面验收标准：用户能确认当前脚本是否已保存、当前版本是什么。

### 3.4 分镜规划中心

- 用户目标：基于脚本稳定维护分镜。
- 当前体验问题：脚本来源、分镜同步关系不清。
- 页面信息结构改动：显示“来源脚本版本”“当前分镜状态”“待同步提示”。
- 组件 / 交互要求：镜头卡、同步提示、冲突提示、批量操作区分层。
- 状态与反馈要求：未保存、已同步、待同步、同步失败必须有清晰文案。
- 依赖的 Runtime 能力：分镜列表、来源版本、同步状态。
- 页面验收标准：用户知道分镜是否落后于脚本。

### 3.5 AI 剪辑工作台

- 用户目标：围绕项目持续完成剪辑与编排。
- 当前体验问题：像孤立工具页，不像创作中枢。
- 页面信息结构改动：首屏固定当前项目、当前时间线、最近任务、下一步动作。
- 组件 / 交互要求：时间线、素材区、任务区、引导区同屏协作，不能互相抢主次。
- 状态与反馈要求：任务处理中、素材缺失、未选择项目、保存失败都必须可见。
- 依赖的 Runtime 能力：时间线版本、素材引用、任务状态、项目上下文。
- 页面验收标准：用户在工作台内知道当前进度和下一步动作。

### 3.6 视频拆解中心

- 用户目标：导入视频并获得可继续使用的拆解结果。
- 当前体验问题：处理中阶段缺少信号，等待感强。
- 页面信息结构改动：拆解流程拆成上传、解析、转写、结构化四段。
- 组件 / 交互要求：阶段条、日志摘要、结果面板、去后续页面入口。
- 状态与反馈要求：可取消、可重试、失败原因可见。
- 依赖的 Runtime 能力：拆解任务、阶段状态、结果摘要。
- 页面验收标准：用户能理解当前卡在哪一步。

#### 3.6.1 实测缺陷（前端侧）

- **Bug F-08 FFprobe 不可用时提示过弱**：用户导入视频后只看到拆解结果里一条静态文案"FFprobe 不可用，已仅记录文件路径和大小"，既没有告知原因也没有修复动作。
- 修复方向：视频拆解页检测到 `error_code = media.ffprobe_unavailable` 时展示醒目黄色警示条，文案含原因（"未检测到 FFprobe 可执行文件"）、影响范围（"时长 / 分辨率 / 码率字段暂无法解析"）、两个动作按钮（"查看修复指引"链接到首启诊断页；"重新扫描"触发后端重探测）；同时在首启向导 / AI 与系统设置 → 诊断区同步一条 FFprobe 健康项，让用户在任一入口都能看到问题。
- 前端验收：首次导入若 FFprobe 不可用，用户能在拆解页看到原因 + 动作按钮，不再是只读降级提示。

### 3.7 配音中心

- 用户目标：把脚本段落稳定生成音轨。
- 当前体验问题：文本、音色、音轨之间缺少上下文绑定。
- 页面信息结构改动：首屏同时展示段落来源、音色选择、当前音轨状态。
- 组件 / 交互要求：试听、重新生成、参数调整、版本说明必须联动。
- 状态与反馈要求：生成中、试听中、生成失败、音轨过期要可见。
- 依赖的 Runtime 能力：音色列表、音轨状态、TTS 任务。
- 页面验收标准：用户能确认当前音轨来自哪版文案和哪套参数。

### 3.8 字幕对齐中心

- 用户目标：完成字幕对齐并确认结果可用。
- 当前体验问题：失败时没有明确修复方向。
- 页面信息结构改动：首屏展示来源音轨、当前字幕版本、对齐状态。
- 组件 / 交互要求：差异摘要、时间轴片段、失败修复入口清晰。
- 状态与反馈要求：对齐中、已对齐、部分失败、需重跑必须可见。
- 依赖的 Runtime 能力：字幕任务、音轨版本、差异信息。
- 页面验收标准：用户知道是重跑对齐还是回到配音中心。

### 3.9 资产中心

- 用户目标：判断资产能否直接用于当前项目。
- 当前体验问题：资产来源与可用性不够明确。
- 页面信息结构改动：优先展示资产状态、来源、最近引用项目。
- 组件 / 交互要求：筛选、预览、引用关系、失效标记统一。
- 状态与反馈要求：生成中、不可用、已失效、已被引用都需明确。
- 依赖的 Runtime 能力：资产元数据、引用关系、生成来源。
- 页面验收标准：用户不会误把不可用资产拖入项目。

### 3.10 账号管理

- 用户目标：判断哪些账号当前可发布。
- 当前体验问题：账号状态不等于用户可理解的发布可用性。
- 页面信息结构改动：优先展示“可发布 / 待验证 / 已失效”。
- 组件 / 交互要求：校验、刷新、失效提示、回执查看入口统一。
- 状态与反馈要求：最近校验时间、失败原因、建议动作明确。
- 依赖的 Runtime 能力：账号状态、验证结果、回执信息。
- 页面验收标准：用户能选出当前可发布账号。

### 3.11 设备与工作区管理

- 用户目标：确认真实执行环境可以用于任务。
- 当前体验问题：设备、工作区、绑定关系混在一起。
- 页面信息结构改动：拆成工作区、执行环境、绑定关系三个视角。
- 组件 / 交互要求：健康检查、目录状态、浏览器实例、绑定关系卡片分层。
- 状态与反馈要求：不可用原因、修复入口、最近检测时间明确。
- 依赖的 Runtime 能力：工作区状态、绑定状态、健康检查结果。
- 页面验收标准：用户能确认哪个真实环境可执行任务。

#### 3.11.1 实测缺陷（前端侧）

- **Bug F-02 浏览器实例永远空态**：`DeviceWorkspaceManagementPage.vue:149-317` 硬编码占位空态（"Runtime 暂时没有返回浏览器实例列表"），页面无新增 / 启动 / 绑定入口，功能实际未实现。
- 修复方向：等后端完成 Bug B-02 的浏览器实例对象模型后，本页补充浏览器实例列表卡片、新建抽屉、启动 / 停止 / 绑定工作区动作；空态文案改为"新建浏览器实例"的引导按钮，不再是被动等待。
- **Bug F-03 按钮布局样式问题**：抽屉底部"取消 / 保存"按钮区、列表操作按钮在紧凑窗口下对齐与宽度不一致，存在错行 / 溢出 / 点击区过小等问题。
- 修复方向：统一使用设计令牌 `--button-gap` / `--drawer-footer-pad`；紧凑窗口下按钮区 `flex-wrap` 并锁定最小高度 44px；同一抽屉的主按钮宽度对齐。
- 前端验收：桌面宽屏与紧凑窗口下按钮不错位不溢出；浏览器实例卡片可新建、启停、绑定，空态带可执行动作而非纯提示。

### 3.12 自动化执行中心

- 用户目标：追踪自动化任务并恢复失败任务。
- 当前体验问题：列表很多，但缺少来源与上下文。
- 页面信息结构改动：突出任务来源、当前状态、最近运行结果。
- 组件 / 交互要求：队列视图、详情抽屉、失败重试入口、日志摘要统一。
- 状态与反馈要求：排队中、运行中、失败、已重试、已取消必须可见。
- 依赖的 Runtime 能力：自动化任务状态、运行记录、日志摘要。
- 页面验收标准：用户能从一个失败任务回到可执行状态。

### 3.13 发布中心

- 用户目标：从预检到提交再到回执完成发布。
- 当前体验问题：当前卡在哪一步不清晰。
- 页面信息结构改动：固定展示预检、提交、回执三段流程。
- 组件 / 交互要求：发布计划、冲突提示、账号设备绑定信息同屏可查。
- 状态与反馈要求：预检失败、提交中、已提交、回执失败必须可见。
- 依赖的 Runtime 能力：预检结果、提交任务、回执状态。
- 页面验收标准：用户能明确知道发布进度与阻塞点。

### 3.14 渲染与导出中心

- 用户目标：确认渲染是否完成且输出文件真实可用。
- 当前体验问题：队列状态和导出结果不够可信。
- 页面信息结构改动：首屏突出任务来源、输出位置、当前阶段。
- 组件 / 交互要求：队列卡、输出详情、失败恢复入口、导出确认区统一。
- 状态与反馈要求：排队、渲染中、导出中、失败、完成可打开要区分。
- 依赖的 Runtime 能力：渲染任务、输出路径、文件存在状态。
- 页面验收标准：用户能确认输出文件是否真的生成。

### 3.15 复盘与优化中心

- 用户目标：把复盘结论转成下一步行动。
- 当前体验问题：复盘更像只读总结，没有行动出口。
- 页面信息结构改动：将建议分成立即处理、下轮处理、仅记录。
- 组件 / 交互要求：问题摘要、建议卡、回流动作按钮分层。
- 状态与反馈要求：生成中、已完成、暂无建议、回流失败都要可见。
- 依赖的 Runtime 能力：复盘摘要、项目历史、回流动作接口。
- 页面验收标准：用户能从复盘中心直接进入下一步修正。

### 3.16 AI 与系统设置

- 用户目标：确认 Provider、模型和系统配置真的可用。
- 当前体验问题：配置是否生效、作用到哪里、失败为什么，都不够清楚。
- 页面信息结构改动：分成 Provider 就绪状态、配置作用范围、诊断结果三层。
- 组件 / 交互要求：测试按钮、作用范围说明、诊断日志摘要、风险提示统一。
- 状态与反馈要求：测试中、可用、部分可用、不可用、配置未保存必须明确。
- 依赖的 Runtime 能力：配置总线、Provider 测试结果、诊断接口。
- 页面验收标准：用户能确认某个 Provider 是否已真正可用。

#### 3.16.1 实测缺陷（前端侧）

- **Bug F-04 "打开日志目录"（已修复）**：旧设置页 composable 已删除，当前由 `SettingsSystemFormPanel.vue` 触发 `open-log-directory`，`AISystemSettingsPage.vue` 调用 `ai-system-settings-page-helpers.ts:openDirectoryPath`，通过 `@tauri-apps/plugin-opener` 的 `openPath` 打开真实本地日志目录。
- 修复方向：新增 `openLogDirectory()` composable，调用 `import('@tauri-apps/plugin-opener').then(m => m.openPath(systemForm.paths.logDir))`；`logDir` 未配置时禁用按钮并提示"请先在路径设置中指定日志目录"；非 Tauri 环境下 fallback 展示路径文本并允许复制。
- **Bug F-05 Provider 启动时缺少静默自检 UI**：配置过 Provider 后每次启动都要手动点"测试"才知道是否可用，桌面壳启动时应让用户无感完成探测，并在 Provider 列表上给出就绪徽标（可用 / 降级 / 不可用）。
- 修复方向：BootstrapGate 启动后调用后端 Bug B-03 提供的 `/api/ai-providers/health` 聚合接口；结果写入 `config-bus` 的 `providerReadiness` map；Provider 管理页标题栏与列表每个 Provider 行显示就绪徽标（颜色 + 最近探测时间 + 重试按钮）；探测失败不阻断启动。
- **Bug F-06 自定义 model_id 无法标注能力类型**：`ai_capabilities.py` schema 已经有 `capabilityTypes: list[str]`，但 Provider 模型新建 / 编辑抽屉没有让用户选择这是 text / vision / video / tts / asset_analysis 的哪一类，导致自定义 `model_id` 无法被能力矩阵识别。
- 修复方向：Provider 模型抽屉新增多选 `capability_kinds`（`text_generation` / `vision` / `video` / `tts` / `asset_analysis`，至少选一项），默认给出合理推断但可修改；保存后在模型列表行内可视化能力标签；能力矩阵页改为从真实 `capability_kinds` 路由而不是用内置元数据猜测。
- **Bug F-07 模型列表可能重复**：保存 Provider 模型时前端没有做 `(provider_id, model_id)` 去重校验，用户重复粘贴会出现同名条目。
- 修复方向：表单 `@blur` 校验当前 Provider 下 `model_id` 是否已存在；已存在则弹中文提示并阻止保存；同时提供"覆盖已有配置"的显式选项，避免用户需要手工删除再新增。
- 前端验收：日志目录按钮一键唤起资源管理器；Provider 列表首屏即可看到就绪徽标；新增自定义 model_id 时必须勾选至少一项能力类型；重复 model_id 保存时有明确中文阻断 + 覆盖选项。

## 4. 前端验收标准

1. 16 页都能回答”当前对象 / 当前状态 / 下一步动作 / 异常反馈”。
2. 阻断态都被替换为可执行引导。
3. 保存、执行、失败、重试的反馈在全站保持统一。
4. Light / Dark 双主题下关键状态都可读、可分辨、可验收。
