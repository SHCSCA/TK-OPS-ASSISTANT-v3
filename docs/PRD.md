## 2. 产品定义与边界

### 2.1 产品定义

TK-OPS 是一个面向内容创作者的 Windows 桌面应用。  
它的核心职责不是替代整个电商后台，而是把视频创作、AI 生成、剪辑编排、配音、字幕、渲染、发布与自动化执行收束到一个可持续演进的本地工作台中。

它应当表现为一个“AI 深度接入的轻量剪辑工作台”：

- 支持从零开始创作视频。
- 支持导入现有视频做拆解、改写、重配音、重字幕与重制。
- 支持在统一时间线中管理 AI 视频片段、音轨、字幕轨和导出任务。
- 支持账号、设备与执行环境管理，但这些能力只服务于创作与发布闭环。
- 支持自动化采集、自动回复、定时发布和任务恢复，但不膨胀成泛后台系统。

### 2.2 核心产品定位

- AI 原生的视频创作中枢。
- 内容创作者的一站式桌面工作台。
- 本地优先、单机授权、深度联网能力按需启用的生产工具。
- 以 TikTok 为第一目标平台的创作与执行系统。

### 2.3 要解决的核心问题

- 内容创作流程被拆散在脚本工具、剪辑工具、AI 工具、表格、浏览器和发布工具中，切换成本高。
- AI 生成、视频剪辑、配音、字幕、发布之间缺乏统一项目上下文。
- 导入已有视频后，难以快速做脚本拆解、镜头重构、配音重做与字幕重对齐。
- 批量视频生产缺少项目级资产管理、任务调度、渲染队列与失败恢复。
- 账号、设备、工作区和浏览器实例之间缺少统一绑定与运行状态追踪。
- AI Provider、模型、音色、字幕策略、提示模板分散，维护成本高。
- 许多桌面工具只做单点功能，无法把“从脚本到成片，再到发布与复盘”的链路真正打通。

### 2.4 V1 范围

V1 必须覆盖以下能力：

- 首启与许可证激活。
- 创作项目总览与最近工程入口。
- 脚本、选题、标题、文案的 AI 生成与改写。
- 分镜规划与镜头组织。
- AI 视频生成、素材组织、时间线编排。
- TTS 配音、音轨管理、字幕生成与对齐。
- 导入视频后的拆解、转写、结构提取与重制入口。
- 资产管理、渲染导出、发布、自动化任务与轻量复盘。
- 账号管理、设备与工作区管理、执行绑定与日志。
- AI Provider、模型、音色、字幕策略、目录与缓存配置。

### 2.5 明确排除项

V1 明确不做以下内容：

- 团队协作、成员权限分工、多人审批流。
- 订单、退款、商品、库存、客服工单等经营后台能力。
- 重 CRM、ERP、店铺经营分析中台。
- 复杂特效、专业调色、重型剪辑插件生态。
- 高风险环境伪装、规避检测、非合规自动化能力。

### 2.6 核心价值

- 一体化：把脚本、镜头、AI 视频、配音、字幕、渲染和发布拉回同一项目上下文。
- AI 深接入：AI 不只是页面按钮，而是贯穿脚本、镜头、素材、声音、字幕和优化建议的系统能力。
- 本地优先：工程文件、缓存、任务、日志留在本地，用户对资产有持续掌控力。
- 可治理：账号、设备、Provider、模型、音色、目录和任务策略统一管理。
- 可演进：V1 采用平衡最优架构，V2 可以逐步升级媒体内核和更重的运行能力。

## 3. 目标用户与典型场景

### 3.1 第一目标用户

#### 3.1.1 内容创作者

- 以单人创作者、小型工作室负责人或极小协作单元为主。
- 关注脚本、镜头、AI 视频、配音、字幕、封面、发布节奏与产能。
- 希望从灵感到成片都在一个系统内完成，不在多个工具来回搬运。
- 需要既支持“从零创作”，也支持“导入现有视频再创作”。

### 3.2 次级用户

#### 3.2.1 自动化执行操作者

- 负责采集、发布、自动回复、队列观察、失败重试与执行健康检查。
- 关注任务可靠性和账号/设备/工作区绑定状态。

#### 3.2.2 个人运营者

- 同时承担创作、发布与简单复盘职责。
- 不需要团队协作后台，但需要项目、账号、设备和自动化能力统一收口。

### 3.3 典型场景

#### 3.3.1 从零创作短视频

用户输入主题、产品卖点或参考方向，系统生成脚本、分镜、镜头说明、配音文本、字幕草稿，随后进入 AI 剪辑工作台完成视频、声音和字幕对齐，再导出与发布。

#### 3.3.2 导入现有视频做拆解重制

用户导入一个现有视频，系统完成转写、段落切分、镜头识别、脚本抽取和字幕恢复，用户在此基础上重写脚本、替换音色、重做字幕并再次生成成片。

#### 3.3.3 批量生产同主题内容

用户以一个项目模板为基准，批量生成多套标题、脚本、镜头组合和音色版本，统一进入渲染队列与发布中心。

#### 3.3.4 账号与设备协同执行

用户把 TikTok 账号绑定到具体设备工作区与浏览器实例，为后续发布、自动回复、采集和健康检查建立明确执行上下文。

#### 3.3.5 轻量复盘驱动下一轮创作

用户查看成片表现、发布状态、执行异常、可复用资产和 AI 给出的下一轮建议，再回流到脚本与分镜页面。

## 4. 核心工作流

### 4.1 首启与授权工作流

`启动应用 -> 创建本地目录 -> 激活许可证 -> 绑定机器指纹 -> 校验 Runtime -> 初始化 AI 配置 -> 进入创作总览`

要求：

- 首次激活必须有清晰成功/失败反馈。
- 授权失败不得直接破坏本地工程数据。
- 授权校验失败时可进入受限模式，但必须保留查看与导出本地项目的能力边界说明。

### 4.2 从零创作工作流

`输入主题/目标 -> 生成脚本 -> 生成分镜 -> 生成视频片段/镜头建议 -> 进入时间线 -> 生成配音 -> 生成字幕 -> 对齐校正 -> 渲染导出 -> 发布`

要求：

- 脚本、分镜、视频、声音、字幕必须能在同一项目对象下追踪。
- 任何 AI 输出都必须支持人工修改、重试、版本保留和回退。
- 时间线是中枢，不是最终只读结果页。

### 4.3 导入拆解重制工作流

`导入视频 -> 识别音轨/字幕 -> 转写与切段 -> 提取脚本结构 -> 恢复镜头节奏 -> 改写脚本 -> 重生成配音/字幕 -> 进入时间线重制 -> 渲染导出`

要求：

- 导入流程必须明确保留原始视频、拆解结果和改写版本。
- 转写、切段、镜头识别失败时要能局部重跑，不强制用户重做整条链路。

### 4.4 执行与发布工作流

`选择项目版本 -> 选择发布账号 -> 选择设备与工作区 -> 发布前校验 -> 进入发布队列 -> 执行 -> 记录结果 -> 失败重试或转人工`

要求：

- 账号、设备、浏览器实例、发布计划必须能建立明确绑定。
- 发布前校验必须展示冲突项、缺失项和风险项。
- 发布与自动化任务必须统一进入任务中心与日志系统。

### 4.5 复盘回流工作流

`查看项目结果 -> 查看渲染/发布/自动化日志 -> 汇总异常与表现 -> 生成下一轮优化建议 -> 回流脚本与分镜`

要求：

- 复盘页要强调可执行建议，而不是堆叠图表。
- 建议必须可直接回流为新项目或项目新版本。

## 6. 页面规格总则

### 6.1 页面结构统一要求

所有正式页面必须包含以下基础信息：

- 页面定位与目标。
- 核心输入。
- 核心输出。
- 后端真实数据来源。
- 核心交互动作。
- 状态与异常反馈。

### 6.2 状态要求

每个页面至少覆盖四类状态：

- 加载中
- 空状态
- 正常状态
- 错误状态

涉及联网、AI 或后台任务的页面，还必须补充：

- 运行中状态
- 队列等待状态
- 重试状态
- 受限模式状态

### 6.3 真实数据要求

- 页面不得长期依赖写死业务数字。
- 页面必须尽量通过 Runtime HTTP / WebSocket 获取真实数据。
- 当真实数据尚不可用时，只允许使用中性空态或明确标注的引导态，不得伪造“看起来真实”的经营指标。

### 6.4 交互与反馈要求

- 所有关键动作必须有进度或结果反馈。
- 所有失败必须给出原因、恢复路径和日志入口。
- 导致全局状态变化的动作必须同步刷新任务、缓存和状态广播。
- 页面内文案统一使用中文业务语义，不使用模糊词。

### 6.5 壳层与局部布局要求

- Sidebar 负责稳定导航，不承载页面内部业务状态。
- Detail Panel 用于上下文详情、属性检查、日志片段或实时预览。
- Status Bar 用于渲染、发布、同步、自动化、AI 运行等持续状态反馈。
- 页面可以使用局部 Tabs，但不得让主工作流跨多个互相割裂的独立页面跳转。

## 7. 16 页逐页规格

### 7.1 `setup_license_wizard`，首启与许可证向导

- 页面目标：完成一机一码激活、本地目录初始化、Runtime 健康检查和首个系统配置。
- 核心输入：许可证信息、基础目录、默认缓存策略、首选 AI Provider。
- 核心输出：可用的本地工作环境与已绑定的许可证状态。
- 关键模块：
  - 许可证激活
  - 机器指纹绑定
  - 目录创建与权限检查
  - Runtime 启动检查
  - 首次配置引导
- 关键规则：
  - 未完成授权前不进入主 Sidebar。
  - 失败时必须保留重试、离线说明和联系支持入口。
  - 激活完成后写入本地安全存储，不在页面层保存敏感密钥。

### 7.2 `creator_dashboard`，创作总览

- 页面目标：作为所有项目和任务的统一入口。
- 核心输入：最近项目、待处理任务、AI 健康状态、渲染状态、发布状态。
- 核心输出：用户下一步动作入口。
- 关键模块：
  - 最近工程卡片
  - 待办与异常提醒
  - AI / 渲染 / 发布健康面板
  - 快速新建项目
  - 最近失败任务回跳
- 关键规则：
  - 不堆叠经营后台图表。
  - 首页重点是“继续工作”和“发现阻塞”。
  - 告警卡必须能跳到具体项目或具体任务。

### 7.3 `script_topic_center`，脚本与选题中心

- 页面目标：完成主题、标题、脚本、文案和版本变体的生成与整理。
- 核心输入：主题、产品卖点、目标人群、时长、风格、参考素材。
- 核心输出：脚本草稿、标题变体、文案版本和可进入分镜的结构化文本。
- 关键模块：
  - 主题与角度面板
  - 标题生成
  - 脚本生成与改写
  - 文案版本管理
  - Prompt 模板与风格预设
- 关键规则：
  - 所有 AI 输出必须支持版本保存与人工编辑。
  - 脚本输出必须能直接流向分镜规划。
  - 不允许把 AI 结果作为不可编辑黑盒。

### 7.4 `storyboard_planning_center`，分镜规划中心

- 页面目标：把脚本结构转为可执行分镜与镜头节奏。
- 核心输入：脚本段落、目标时长、镜头风格、素材来源、节奏模板。
- 核心输出：镜头列表、镜头说明、节奏结构和可送往剪辑工作台的镜头计划。
- 关键模块：
  - 脚本段落映射
  - 分镜卡片与顺序调整
  - 镜头提示词
  - 过场与节奏建议
  - 版本对比
- 关键规则：
  - 分镜必须和脚本段落建立双向引用。
  - 镜头计划应能同时服务 AI 视频生成和素材拼装两条路径。
  - 修改脚本后应提示是否同步刷新分镜。

### 7.5 `ai_editing_workspace`，AI 剪辑工作台

- 页面目标：成为视频、音轨、字幕和 AI 输出的统一时间线工作台。
- 核心输入：分镜计划、视频片段、音频素材、字幕轨、AI 生成片段、导入素材。
- 核心输出：可渲染的完整时间线工程。
- 关键模块：
  - 多轨时间线
  - 预览窗口
  - 镜头/片段面板
  - AI 视频片段生成与替换
  - 音轨与字幕轨管理
  - Detail Panel 属性编辑
- 关键规则：
  - 这是产品核心页，V1 要做到“像轻量剪映，但不做复杂特效”。
  - 必须支持在时间线内直接调用 AI 生成视频、声音和字幕。
  - 视频、声音、字幕必须可在同一上下文中对齐。
  - 任何长任务都应后台执行，不阻塞编辑工作流。

### 7.6 `video_deconstruction_center`，视频拆解中心

- 页面目标：把导入视频转为可重制项目。
- 核心输入：原始视频文件、导入来源、语言信息、拆解策略。
- 核心输出：转写文本、切段结果、镜头摘要、脚本结构和重制草稿。
- 关键模块：
  - 导入队列
  - 转写与切段
  - 镜头识别
  - 原字幕恢复
  - 脚本提取与改写入口
- 关键规则：
  - 原视频、拆解结果和重制版本必须分层保存。
  - 任一步骤失败时允许局部重跑。
  - 拆解结果必须能直接回流脚本、分镜和剪辑工作台。

### 7.7 `voice_studio`，配音中心

- 页面目标：完成 TTS 配音、音色选择、语速停顿调节和配音版本管理。
- 核心输入：脚本文本、角色设定、音色、情绪、语言、段落切分。
- 核心输出：音轨文件、段落级时间信息和可进入时间线的 VoiceTrack。
- 关键模块：
  - 角色音色库
  - TTS 参数面板
  - 试听与对比
  - 段落切分
  - 版本管理
- 关键规则：
  - 配音结果必须保留段落级映射，供字幕对齐与时间线编辑使用。
  - 支持多音色候选，但默认只保留有限数量的高价值版本。
  - 所有音频生成任务进入统一任务中心。

### 7.8 `subtitle_alignment_center`，字幕对齐中心

- 页面目标：生成、编辑、校正并导出与音视频对齐的字幕轨。
- 核心输入：脚本文本、音轨、转写结果、时间线信息、字幕模板。
- 核心输出：SubtitleTrack、字幕样式配置和导出文件。
- 关键模块：
  - 自动生成字幕
  - 时间码校正
  - 逐段文本编辑
  - 样式模板
  - 时间线联动预览
- 关键规则：
  - 字幕必须可回写到时间线。
  - 自动对齐失败时必须支持局部手调。
  - 字幕模板属于资产中心管理对象，不在页面本地孤立保存。

### 7.9 `asset_library`，资产中心

- 页面目标：集中管理工程资产与可复用资源。
- 核心输入：视频片段、图片、BGM、音频、字幕模板、封面模板、工程版本。
- 核心输出：可检索、可引用、可回溯的资产集合。
- 关键模块：
  - 资产列表
  - 标签与分组
  - 引用关系
  - 版本与来源
  - 预览与替换
- 关键规则：
  - 每个资产应记录来源、用途、创建时间和引用项目。
  - 工程资产与全局资产应做命名和范围隔离。
  - 删除资产前必须提示影响范围。

### 7.10 `account_management`，账号管理

- 页面目标：管理 TikTok 账号资料、分组、状态、发布目标与执行绑定关系。
- 核心输入：账号名称、平台信息、分组标签、状态、绑定设备、发布策略。
- 核心输出：可用于发布与自动化的 Account 记录。
- 关键模块：
  - 账号列表
  - 分组与标签
  - 状态与健康
  - 绑定关系
  - 发布目标配置
- 关键规则：
  - 账号管理只服务内容发布与执行，不扩展到订单、退款、商品后台。
  - 账号状态变化必须同步影响发布中心与自动化中心。
  - 敏感信息默认脱敏展示。

### 7.11 `device_workspace_management`，设备与工作区管理

- 页面目标：管理真实 PC 工作区、浏览器实例与执行环境。
- 核心输入：工作区名称、浏览器实例、目录、网络状态、运行日志、绑定账号。
- 核心输出：可审计、可切换、可恢复的 DeviceWorkspace 记录。
- 关键模块：
  - 工作区列表
  - 浏览器实例管理
  - 执行环境健康检查
  - 账号绑定
  - 日志与诊断
- 关键规则：
  - 只管理真实工作区与执行环境，不提供高风险环境伪装能力。
  - 设备状态必须可用于发布预检与自动化任务调度。
  - 每个工作区应记录最后使用时间、异常次数和绑定关系。

### 7.12 `automation_console`，自动化执行中心

- 页面目标：统一管理采集、自动回复、同步、校验和失败恢复任务。
- 核心输入：任务类型、触发规则、账号绑定、设备工作区、时间计划、重试策略。
- 核心输出：AutomationTask 队列、日志与执行结果。
- 关键模块：
  - 任务列表
  - 规则与计划
  - 运行日志
  - 异常重试
  - 健康状态
- 关键规则：
  - 自动化任务必须进入统一调度和日志系统。
  - 失败原因必须明确可见。
  - 自动回复能力必须可配置开关、范围和人工兜底策略。

### 7.13 `publishing_center`，发布中心

- 页面目标：完成发布计划、定时发布、发布前校验和结果追踪。
- 核心输入：项目版本、目标账号、目标设备、发布时间、发布模板、备注。
- 核心输出：PublishPlan、发布任务记录和回执状态。
- 关键模块：
  - 待发布列表
  - 日历与排期
  - 预检清单
  - 发布回执
  - 冲突检测
- 关键规则：
  - 发布前必须校验账号、设备、工作区、素材、字幕和时长等关键条件。
  - 发布失败时必须支持转人工或重试。
  - 发布能力应优先走合规路径，必要时提供手动确认模式。

### 7.14 `render_export_center`，渲染与导出中心

- 页面目标：统一管理渲染、转码、批量导出和文件输出。
- 核心输入：Timeline、导出模板、分辨率、码率、字幕开关、音轨策略。
- 核心输出：RenderTask、导出文件和失败记录。
- 关键模块：
  - 渲染队列
  - 输出配置
  - 资源占用监控
  - 失败恢复
  - 导出历史
- 关键规则：
  - 渲染任务必须支持排队、取消、重试和错误定位。
  - 导出路径与命名规则必须可配置并接入统一配置总线。
  - 渲染成功后应可直接流向发布中心。

### 7.15 `review_optimization_center`，复盘与优化中心

- 页面目标：把结果回看、问题定位和下一轮创作建议集中到同一页。
- 核心输入：项目结果、发布状态、任务日志、资产使用情况、异常记录。
- 核心输出：可执行优化建议与新版本入口。
- 关键模块：
  - 项目结果摘要
  - 异常与阻塞记录
  - 可复用资产提示
  - AI 优化建议
  - 回流新项目入口
- 关键规则：
  - 复盘页以行动建议为主，不做重经营中台。
  - 建议必须能回流到脚本、分镜或时间线新版本。
  - 日志和结果要能互相跳转。

### 7.16 `ai_system_settings`，AI 与系统设置

- 页面目标：统一管理 AI Provider、模型、音色、字幕策略、目录、缓存、日志与系统参数。
- 核心输入：Provider 配置、模型策略、Fallback 规则、音色、路径、缓存、日志级别。
- 核心输出：可被全局读取的系统配置与健康状态。
- 关键模块：
  - Provider 管理
  - 模型管理
  - 音色与配音参数
  - 字幕策略
  - 目录与缓存
  - 日志与诊断
- 关键规则：
  - 所有系统配置统一走配置总线，不允许页面、脚本、服务各自保存一套配置。
  - Provider 测试结果必须有明确成功/失败反馈。
  - 密钥进入安全存储，不明文暴露在日志和页面上。

## 8. 技术选型与系统架构

### 8.1 技术选型

| 层级 | 选型 | 说明 |
| --- | --- | --- |
| 桌面壳 | Tauri 2 | 提供 Windows 桌面宿主与系统级能力 |
| 前端 | Vue 3 + TypeScript + Vite | 提供高交互工作台和页面渲染能力 |
| 状态与路由 | Pinia + Vue Router | 负责全局状态、页面流转与模块化管理 |
| Python Runtime | FastAPI | 提供本地 HTTP / WebSocket 服务与业务编排 |
| 数据访问 | SQLAlchemy + Alembic | 提供实体映射、迁移与持久化能力 |
| 本地数据库 | SQLite | 默认单机落盘，便于本地优先运行 |
| 媒体流水线 | FFmpeg 类工具链 | 负责转码、拼接、音视频处理与导出 |
| AI 能力层 | 多 Provider 适配层 | 统一脚本、视频、TTS、字幕和分析能力 |

### 8.2 架构原则

- V1 采用平衡最优架构，不为更重媒体内核过度设计。
- 前端负责交互、时间线编辑与状态表达，Python Runtime 负责业务编排、任务执行与持久化。
- AI、渲染、发布、自动化任务必须在后台执行，不阻塞主界面。
- 任何系统能力都应通过明确接口接入，不允许跨层直接调用散乱实现。

### 8.3 系统分层

- 表现层：Tauri + Vue 页面、App Shell、时间线编辑与状态展示。
- 应用层：项目编排、AI 工作流、发布预检、任务调度、渲染编排。
- 领域层：Project、Script、Storyboard、Timeline、Account、DeviceWorkspace、RenderTask 等核心模型。
- 基础设施层：SQLite、文件系统、日志、密钥存储、FFmpeg、外部 AI Provider、网络通信。

### 8.4 壳层与运行时边界

- 桌面端所有业务数据优先通过 Runtime API 获取。
- 前端与 Runtime 的主通信方式为 HTTP + WebSocket。
- 前端不得回退到旧壳桥接模式。
- 旧 PySide6 壳不进入运行、联调、打包或发布路径。

### 8.5 Runtime 契约

Runtime 返回统一 JSON 信封：

```json
{ "ok": true, "data": {} }
```

```json
{ "ok": false, "error": "错误说明" }
```

要求：

- 错误信息必须可直接用于 UI 中文提示。
- WebSocket 事件至少覆盖任务进度、任务日志、任务完成、任务失败、系统健康变化。
- 前端收到数据变更后必须刷新相关缓存、任务列表与状态广播。

### 8.6 媒体流水线定位

- 媒体流水线是独立系统能力，不内嵌在页面局部逻辑中。
- V1 重点覆盖转码、拼接、音视频轨对齐、字幕烧录/外挂、封面输出和批量导出。
- V1 不做复杂特效引擎，但必须为时间线、音轨、字幕轨和片段替换提供可靠底座。
- V2 可在不推翻前端工作台的前提下升级更重的媒体内核能力。

### 8.7 全局系统能力

- ConfigBus：统一广播配置变更与持久化。
- Task Queue：统一接收 AI、渲染、发布、自动化等长任务。
- Health Monitor：统一展示 Provider、Runtime、设备和任务运行状态。
- Audit Log：统一记录关键操作、失败与恢复行为。

## 9. AI 能力体系

### 9.1 AI 定位

AI 在 TK-OPS 中是系统能力，不是单页附属功能。  
它必须深度参与以下环节：

- 主题与脚本生成
- 标题与文案变体
- 分镜规划
- 视频拆解与结构识别
- AI 视频片段生成
- TTS 配音
- 字幕生成与对齐
- 资产分析与复盘建议

### 9.2 Provider 能力分层

V1 的 AI 适配层按能力接口分层：

- `ScriptProvider`
- `VideoGenerationProvider`
- `TTSProvider`
- `SubtitleAlignmentProvider`
- `AssetAnalysisProvider`

要求：

- Provider 能力通过统一协议接入，不在页面层写死供应商逻辑。
- 同一页面可以调用多个能力接口，但调用链必须被记录和可回溯。

### 9.3 双层 AI 入口

AI 必须同时以两种方式存在：

- 页面内 AI 助手：
  - 在脚本、分镜、配音、字幕、复盘等页提供上下文增强能力。
- 中央 AI 工作流：
  - 在 AI 剪辑工作台和后台任务中心承接批量生成、重跑、异步执行和版本化结果。

### 9.4 模型治理

- Provider 可切换。
- 模型可切换。
- 每类能力支持推荐默认模型。
- 支持回退链路，但必须记录使用了哪个 Provider、哪个模型以及失败原因。
- AI 调用日志至少记录：
  - 能力类型
  - Provider
  - 模型
  - 耗时
  - 成本估计
  - 结果状态

### 9.5 Prompt 与预设治理

- Prompt 模板与风格预设必须统一管理。
- 模板支持版本化与回滚。
- 页面只消费模板，不在组件内硬编码大段提示词。
- 创作语气、脚本风格、镜头风格、音色偏好和字幕策略都应可配置。

### 9.6 输出治理

- 所有 AI 输出都必须支持人工校正。
- 所有 AI 输出都必须保留来源与版本。
- 无法稳定生成时必须给出可见的失败原因和重试建议。
- AI 输出不应直接覆盖用户已编辑内容，必须明确采用“新版本”或“覆盖确认”路径。

## 10. 账号、设备与自动化体系

### 10.1 账号体系

账号体系只服务于创作执行闭环，至少覆盖：

- 账号基础信息
- 分组与标签
- 账号状态
- 发布目标
- 绑定设备与工作区
- 自动化规则关联

不包含：

- 订单管理
- 商品管理
- 退款处理
- 团队客服后台

### 10.2 设备与工作区体系

设备体系的核心对象是 `DeviceWorkspace`，而不是抽象环境壳。

V1 至少覆盖：

- 本地工作区目录
- 浏览器实例
- 执行环境状态
- 日志与异常记录
- 账号绑定
- 最后使用与健康检查

要求：

- 设备与工作区必须真实存在、可审计、可恢复。
- 不定义高风险伪装能力。

### 10.3 执行绑定

`ExecutionBinding` 用于描述三类对象关系：

- Account
- DeviceWorkspace
- 执行任务

它的作用是确保：

- 发布前能做完整预检。
- 自动化任务能找到明确执行上下文。
- 故障后能快速回溯是哪一个账号、哪一个工作区、哪一个任务失败。

### 10.4 自动化任务

V1 自动化任务至少分为：

- 采集任务
- 发布任务
- 回复任务
- 渲染任务
- 同步/校验任务

所有自动化任务必须支持：

- 创建
- 排队
- 运行
- 失败原因展示
- 手动重试
- 取消
- 日志查看

## 11. 数据模型与任务模型

### 11.1 核心项目模型

V1 固定使用以下项目主链：

`Project -> Script -> Storyboard -> Timeline -> VoiceTrack -> SubtitleTrack -> RenderTask`

要求：

- 任何创作路径都必须回收到同一 Project。
- 从零创作和导入拆解只是入口不同，不允许形成两套互不兼容的数据模型。

### 11.2 关键实体

| 实体 | 说明 |
| --- | --- |
| `Project` | 项目根对象，聚合脚本、分镜、时间线、任务和版本 |
| `Script` | 脚本正文、标题、文案、版本与来源 |
| `Storyboard` | 镜头规划、节奏、视觉说明与脚本映射 |
| `Timeline` | 多轨时间线工程，管理视频、音频、字幕与导出配置 |
| `VoiceTrack` | TTS 或导入音频的结构化音轨 |
| `SubtitleTrack` | 字幕段、时间码、样式与版本 |
| `Asset` | 视频、图片、音频、封面、模板等资产 |
| `Account` | TikTok 账号对象 |
| `DeviceWorkspace` | PC 工作区、浏览器实例与执行环境对象 |
| `ExecutionBinding` | 账号、工作区与任务的绑定关系 |
| `PublishPlan` | 发布时间、目标账号、预检结果与回执 |
| `AutomationTask` | 采集、回复、同步、校验等任务对象 |
| `RenderTask` | 渲染、转码、导出对象 |
| `LicenseGrant` | 授权状态、绑定信息与校验记录 |
| `AIJobRecord` | AI 调用记录、模型、结果状态与耗时 |

### 11.3 任务状态模型

所有长任务统一采用状态机：

- `draft`
- `queued`
- `running`
- `blocked`
- `succeeded`
- `failed`
- `cancelled`

要求：

- 任务失败必须带错误信息、日志片段和重试入口。
- 支持区分“可自动重试失败”和“必须人工处理失败”。

### 11.4 本地目录与落盘

默认数据根目录为：

- `%APPDATA%/TK-OPS-ASSISTANT/`

建议至少包含：

- `db/`
- `projects/`
- `assets/`
- `renders/`
- `logs/`
- `cache/`
- `licenses/`

### 11.5 配置与密钥

- 所有配置统一通过配置总线读取与写入。
- 密钥和许可证信息必须进入安全存储或受控位置。
- 页面不得在本地随意写入独立配置文件。

## 12. 工程硬约束

### 12.1 语言与交互

- 全局文案、交互提示、注释优先中文。
- 新增注释必须使用中文，并保持简洁，只解释意图、边界和复杂逻辑。

### 12.2 真实数据与真实交互

- 页面必须尽量由后端真实数据驱动。
- 禁止把可点击原型当最终结果。
- 页面数据变化必须有真实 Runtime 路径，不允许长期停留在静态假数据。

### 12.3 目录边界与全局一致性

- 新功能默认只落在 `apps/desktop` 与 `apps/py-runtime`。
- 所有实现必须遵循统一目录边界、统一架构、统一规范。
- 页面、数据层、帮助函数、类型、样式与测试必须按职责拆分。

### 12.4 禁止大文件

- 不允许继续出现单文件数千行实现。
- 页面应拆分为页面壳、composable、helpers、types、styles、runtime adapter 等小文件。

### 12.5 全局异常与日志

- 所有异常必须被捕获、记录并转换为可见反馈。
- 日志格式必须统一，至少覆盖：
  - system
  - business
  - task
  - ai
  - audit

### 12.6 全局配置总线

- 所有配置必须经过统一配置入口或配置总线管理。
- 禁止页面、脚本、服务各自保存一套配置。
- 配置变更必须可广播、可持久化、可追踪。

### 12.7 单壳运行约束

- 旧 PySide6 壳只允许作为迁移参考。
- 旧壳不得进入运行、联调、打包、灰度和发布路径。
- 新链路必须统一通过 Tauri 桌面壳 + Python Runtime 实现。

### 12.8 接口与调用约束

- 前后端统一通过 Runtime HTTP / WebSocket 通信。
- Runtime 返回统一 JSON 信封。
- 页面不得绕过服务层直接拼接业务规则。

## 13. 非功能要求

### 13.1 性能

- 首次启动应在合理时间内完成首屏展示。
- 二次启动应明显快于首次启动。
- 页面切换不应阻塞主线程。
- 时间线编辑、素材列表和任务列表必须保持可接受的滚动与操作流畅度。
- AI 流式输出应支持边生成边展示。

### 13.2 稳定性

- 所有长任务必须后台执行。
- 关键任务必须可重试。
- 应用异常退出后应尽量恢复最近项目与任务上下文。
- 单个任务失败不得拖垮整个桌面壳。

### 13.3 安全性

- 许可证、API Key 与账号敏感信息必须受控存储。
- 敏感信息默认脱敏展示。
- 日志不得暴露密钥和敏感标识。
- 发布、覆盖、删除、批量任务等高影响操作必须有二次确认或明确风险提示。

### 13.4 可观测性

- 必须有系统日志、任务日志、AI 调用日志、设备日志与发布日志。
- 必须能追踪任务耗时、失败率、重试次数和当前队列状态。
- 必须能观察 Provider 健康、Runtime 健康与设备健康。

### 13.5 可维护性

- 页面逻辑、领域逻辑与基础设施逻辑必须分离。
- 关键系统能力通过接口管理，便于替换和扩展。
- 文档、页面结构、Runtime 契约与配置模型必须保持一致。

### 13.6 可扩展性

- 新 AI Provider 可通过能力接口接入。
- 新页面可通过统一路由与模块规范接入。
- 新导出模板、字幕模板、音色模板可作为资产或配置接入。
- V2 可以在不推翻 V1 工作台的前提下升级更重的媒体内核。

### 14.4 主要风险与应对

#### 14.4.1 AI Provider 不稳定

- 多 Provider 能力抽象。
- 关键能力支持模型回退。
- 失败结果明确回写日志和任务状态。

#### 14.4.2 时间线工作台复杂度过高

- V1 聚焦 AI 深接入和基础编排，不做复杂特效。
- 先把镜头、音轨、字幕轨与预览链路做稳。

#### 14.4.3 自动化能力边界失控

- 自动化从属于创作中枢，不独立膨胀成后台系统。
- 高风险能力不进入产品定义。

#### 14.4.4 本地优先带来的环境差异

- 首启向导承担目录、权限、Runtime 与依赖检查。
- 关键目录与配置全部可见、可诊断、可修复。

- 本文作为唯一真源。
- 所有新增模块、接口和页面必须回到本文对应章节核对。

## 15. 文档验收场景与完成标准

### 15.1 验收场景

新版 PRD 完成后，至少应能用以下场景检查其可执行性：

- 首启场景：完整定义一机一码激活、目录初始化、Runtime 准备和失败反馈。
- 从零创作场景：完整覆盖脚本、分镜、视频生成、配音、字幕、时间线成片。
- 导入拆解场景：完整覆盖导入视频、转写、结构拆解、改写与重制回流。
- 配音字幕场景：明确 TTS、音轨、字幕对齐、时间码编辑和错误恢复。
- 账号设备场景：明确账号、工作区、执行环境、绑定关系、健康状态和日志。
- 自动化场景：明确采集、自动回复、定时发布、队列、重试和冲突检查。
- 渲染导出场景：明确渲染任务、导出、失败恢复和资源占用反馈。
- 复盘场景：定义轻量复盘如何回流到下一轮创作。

### 15.2 完成标准

- 产品叙事完全切换为 AI 视频创作中枢。
- 页面树固定为 16 页平衡方案，不再回退到旧 26 / 27 / 34 / 35 页口径。
- 技术主线完全切换为 Tauri + Vue + Python Runtime。
- 账号管理与设备管理被写成核心支撑域，而不是外挂工具页。
- AI 能力、数据模型、任务模型、工程硬约束均被写成可直接落地的明确要求。
- 文档中不再把旧企业后台、旧团队协作、旧 Qt 运行主线作为当前目标。

## 16. 结论

TK-OPS 的当前版本不是“TikTok Shop 团队运营后台”，而是一个面向内容创作者的 AI 视频创作中枢。  
它的第一性目标是把脚本、分镜、AI 视频、TTS、字幕、时间线、渲染、发布和自动化执行统一到同一个本地优先的桌面工作台中，并通过账号管理、设备与工作区管理、统一配置总线、任务队列与日志体系提供稳定支撑。

本文自此作为产品唯一真源，后续设计与实现均应以本文定义的 16 页结构、技术主线、模型边界和工程硬约束为准。


## 17. V2 体验问题与改造目标

### 17.1 V2 问题来源

当前 V2 文档包来自真实用户体验评估，核心问题固定为：

1. 被“阻断”替代引导
2. 页面完成度不一致
3. 文案、结构、行为不一致
4. 核心创作闭环不顺
5. 用户信任感不足

### 17.2 V2 改造目标

- 把阻断改造成可执行引导
- 把 16 页收口到统一闭环
- 把保存、执行、失败、重试反馈标准化
- 把用户信任建立在真实数据、真实任务和可确认结果上

## 18. 核心闭环质量要求

1. 首启、授权、目录初始化、Runtime 健康检查必须可见且可恢复。
2. 项目进入后，脚本、分镜、配音、字幕、剪辑、渲染、发布、复盘必须保持同一项目上下文。
3. 所有关键写操作必须能确认是否保存成功。
4. 所有长任务必须有统一任务状态、失败原因与重试路径。
