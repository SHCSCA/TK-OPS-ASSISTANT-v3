# 2026-04-25 火山方舟旧模型别名修复设计

## 问题定义

火山方舟远端模型目录返回的模型 ID 已经版本化，例如：

- `doubao-seed-1-6-250615`
- `doubao-seed-1-6-251015`

但系统静态目录仍存在旧别名：

- `doubao-seed-1.6`

因此会出现两类错误：

1. 设置页仍可能展示或继续使用旧别名。
2. 健康检查、文本生成等主链路把旧别名传给远端，最终得到 `404 does not exist`。

## 设计决策

### 1. 统一模型别名解析

在 `AICapabilityService` 内新增统一解析逻辑：

- 输入：`provider_id + model_id + 可选能力类型`
- 输出：真实可用的模型 ID

解析规则：

- 优先返回启用中的精确模型 ID。
- 若精确 ID 不可用，则按归一化后的模型族前缀匹配远端模型。
- 当存在多个候选时，优先选择同能力类型的候选，并选择排序上最新的版本化模型。

### 2. 模型目录去重

对支持远端模型同步的 Provider：

- 当远端已存在某个模型族的真实模型时，隐藏同族的静态旧别名。
- 这样前端模型目录和健康检查下拉框都只看到真实模型。

### 3. 运行时调用统一使用解析结果

以下链路统一改用解析后的模型 ID：

- Provider 健康检查
- 文本生成请求

这样即使历史配置里保存了旧别名，运行时也能自动走到真实模型。

## 验证标准

- 火山方舟完成远端同步后，`/api/settings/ai-providers/volcengine/models` 不再返回 `doubao-seed-1.6`。
- 对旧别名做健康检查时，实际探测模型应解析为最新可用版本化模型。
- 文本生成服务调用 Provider 时，请求体里的模型 ID 应为解析后的真实模型。
