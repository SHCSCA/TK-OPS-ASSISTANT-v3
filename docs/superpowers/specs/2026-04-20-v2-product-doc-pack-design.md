# TK-OPS V2 产品文档包设计

## 1. 设计目标

本设计用于锁定 V2 文档包的结构、术语和真源映射策略，确保本轮输出是一组可直接驱动前后端收口的产品需求文档，而不是一次松散的体验点评记录。

## 2. 文档包结构

### 2.1 总文档

`docs/V2-PRODUCT-REQUIREMENTS-2026-04-20.md`

职责：

- 作为 V2 问题与需求总入口
- 统一定义问题摘要、业务影响与 V2 目标
- 提供 16 页问题矩阵
- 定义核心闭环 V2 目标

### 2.2 前端文档

`docs/V2-FRONTEND-REQUIREMENTS-2026-04-20.md`

职责：

- 定义跨页面交互一致性规则
- 定义 16 页逐页前端需求
- 明确页面结构、组件、状态、反馈与验收

### 2.3 后端文档

`docs/V2-BACKEND-REQUIREMENTS-2026-04-20.md`

职责：

- 定义跨页面契约与状态规则
- 定义 16 页逐页后端需求
- 明确接口补齐、错误码、广播、保存结果、可追踪性与验收

## 3. 固定术语口径

### 3.1 阻断

指用户当前无法继续，但原因、范围、下一步与可执行动作尚不充分的状态。  
V2 目标是把“阻断”改造成“可执行引导”。

### 3.2 引导

指系统明确告诉用户：

- 为什么当前不能继续
- 受影响范围是什么
- 需要补什么条件
- 现在可以点什么
- 完成后会发生什么

### 3.3 任务状态

统一指长任务或异步操作的可见状态链路，包括：

- 已开始
- 处理中
- 成功
- 失败
- 可重试
- 已取消

### 3.4 真实数据

指来源于真实 Runtime、真实任务、真实配置、真实本地对象或真实 AI 调用结果的数据。  
无真实数据时使用中性空态，不使用伪业务数字。

### 3.5 配置总线

指所有页面、脚本、服务统一读取和写入的配置入口。  
V2 文档中的配置依赖必须归到统一配置入口，不允许散落表述。

### 3.6 JSON 信封

统一指 Runtime 返回信封：

- 成功：`{ "ok": true, "data": ... }`
- 失败：`{ "ok": false, "error": "..." }`

## 4. 真源映射策略

### 4.1 PRD

只补充产品级问题、V2 改造目标、核心闭环质量要求与文档入口，不复制 16 页逐项需求全文。

### 4.2 UI-DESIGN-PRD

只补充交互一致性规则、阻断与引导规则、执行反馈规则与重点页面优先原则。

### 4.3 PROJECT-STATUS

只说明当前状态已经进入“已有实现 + V2 文档已建档 + 待按 V2 收口”。

### 4.4 历史后端需求文档

只标注其历史定位并链接到新的 V2 文档入口，防止后续误用。

## 5. 内容边界

### 5.1 本轮包含

- 计划文档
- 设计文档
- V2 总文档
- V2 前端需求文档
- V2 后端需求文档
- 现有真源入口回写

### 5.2 本轮不包含

- 代码实现
- 路由或模型改造
- 直接修改 `docs/RUNTIME-API-CALLS.md`
- 新增与 16 页范围无关的产品扩页

## 6. 质量要求

### 6.1 结构质量

- 各文档章节命名清晰，职责不重叠。
- 16 页内容全覆盖，格式统一。

### 6.2 执行质量

- 每页都必须落到“问题 / 需求 / 依赖 / 验收”。
- 避免“优化体验”“提升一致性”这类空泛表述。

### 6.3 一致性质量

- 前端、后端、产品文档的术语口径一致。
- 真源入口与 V2 包之间的链接关系清楚可查。
