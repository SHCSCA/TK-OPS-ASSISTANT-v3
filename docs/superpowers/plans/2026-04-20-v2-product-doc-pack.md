# TK-OPS V2 产品文档包计划

## 1. 任务定位

本计划是一次文档治理任务，不进入代码实现，不直接修改 Runtime API 真源字段清单。  
目标是把“真实用户体验评估”沉淀为一组可执行、可交付、可验收的 V2 需求文档，并把入口同步回写到现有真源文档。

## 2. 问题来源

本轮文档以真实用户体验评估为来源，核心问题固定为：

1. 被“阻断”替代引导
2. 页面完成度不一致
3. 文案、结构、行为不一致
4. 核心创作闭环不顺
5. 用户信任感不足

这些问题必须统一转写为：

- 用户感知问题
- 业务影响
- V2 改造目标
- 前端需求
- 后端需求
- 验收标准

## 3. 交付物

本次必须新增以下文件：

1. `docs/superpowers/plans/2026-04-20-v2-product-doc-pack.md`
2. `docs/superpowers/specs/2026-04-20-v2-product-doc-pack-design.md`
3. `docs/V2-PRODUCT-REQUIREMENTS-2026-04-20.md`
4. `docs/V2-FRONTEND-REQUIREMENTS-2026-04-20.md`
5. `docs/V2-BACKEND-REQUIREMENTS-2026-04-20.md`

并同步回写：

1. `docs/PRD.md`
2. `docs/UI-DESIGN-PRD.md`
3. `docs/PROJECT-STATUS.md`
4. `docs/BACKEND-REQUIREMENTS-2026-04-17.md`

## 4. 文档结构原则

### 4.1 独立文档包

- V2 需求不覆盖现有 PRD 与 UI 真源全文。
- V2 采用独立文件名保留演进轨迹。
- 真源文档只承担原则、入口与定位说明，不复制逐页全文。

### 4.2 1 主 2 分

- `V2-PRODUCT`：总入口，负责问题摘要、全局原则、16 页矩阵、核心闭环目标。
- `V2-FRONTEND`：负责跨页前端规则与 16 页逐页前端需求。
- `V2-BACKEND`：负责跨页后端规则与 16 页逐页后端需求。

### 4.3 全量 16 页

本轮范围固定覆盖当前正式产品的 16 页，不只写 P0/P1 页面，不留“后续再补”的结构性空白。

### 4.4 只写需求，不写实现方案

- 文档写到前后端 implementer 无需再猜页面或接口应如何表现。
- 文档不直接写代码方案，不替代 API 真源和设计稿。

## 5. 实施步骤

### 5.1 计划与设计补档

- 新增 plan 文档，锁定任务是文档治理。
- 新增 spec 文档，锁定章节结构、术语口径、回写策略。

### 5.2 总文档成包

- 形成 V2 总文档。
- 将五类体验问题转成产品经理口径的 V2 目标。
- 固定 16 页问题矩阵结构。

### 5.3 前后端拆分

- 前端文档按“跨页规则 + 逐页需求”编写。
- 后端文档按“契约、状态、广播、可追踪性”编写。

### 5.4 真源回写

- PRD 增补 V2 体验问题与改造目标、核心闭环质量要求、V2 入口。
- UI 真源补充 V2 交互一致性规则。
- 项目状态文档更新为“已有实现 + V2 文档建档 + 待按 V2 收口”。
- 历史后端需求文档显式标注历史定位，并指向新 V2 文档。

## 6. 验收标准

### 6.1 结构验收

- 三份 V2 文档全部存在。
- 总文档包含 16 页矩阵。
- 前端文档包含跨页规则与 16 页逐页需求。
- 后端文档包含跨页规则、共性章节与 16 页逐页需求。

### 6.2 文本验收

- 标题、日期、V2 命名统一。
- 术语统一使用“阻断 / 引导 / 任务状态 / 真实数据 / 配置总线 / JSON 信封”。
- 全文保持中文，不出现关键章节空话。

### 6.3 真源映射验收

- `docs/PRD.md`、`docs/UI-DESIGN-PRD.md`、`docs/PROJECT-STATUS.md`、`docs/BACKEND-REQUIREMENTS-2026-04-17.md` 都能明确指向新的 V2 文档包。
- 不出现相互冲突的“下一阶段需求入口”。
