## 2. 文档目标

本 PRD 用于统一 TK-OPS 的视觉语言、壳层结构、页面布局、组件标准和主题规范，确保产品设计与实现遵循同一套界面体系。

本文的目标不是描述一个泛后台 UI，而是描述一个面向内容创作者的 AI 视频创作中枢界面系统。文档必须服务以下结果：

1. 全局外壳稳定，用户切换页面时不需要重新学习导航。
2. AI 剪辑工作台、脚本、分镜、配音、字幕、渲染、发布和自动化页面共享一套设计系统。
3. Light / Dark 双主题在不改变页面结构的前提下切换。
4. 所有颜色、字体、间距、圆角、阴影、动效都可映射为前端可执行的设计令牌。
5. 16 个正式页面都有统一的设计规范与落地边界。
6. 页面必须优先面向真实数据、真实任务和真实运行状态，而不是可点击原型。

## 3. 产品与界面定位

### 3.1 产品定位

TK-OPS 是一个 AI 视频创作中枢，不是团队协作后台，也不是经营系统。

UI 的核心职责是：

- 帮助内容创作者快速进入项目。
- 让脚本、分镜、视频、声音、字幕、任务、发布和复盘共享同一上下文。
- 在复杂任务运行时保持状态透明、错误可见、恢复路径明确。
- 让 AI 深度参与工作流，但不把用户锁进不可理解的黑盒。

### 3.2 第一目标用户

1. 内容创作者。
2. 自动化执行操作者。
3. 个人运营者。

### 3.3 产品气质

1. 专业，不浮夸。
2. 创作优先，不后台化。
3. 高信息密度，但阅读压力低。
4. AI 深接入，但所有结果都可解释、可编辑、可回退。
5. 桌面优先，强调多栏布局、持续状态栏和任务并行感知。

## 4. 设计理念

### 4.1 壳层稳定，工作流连续

TK-OPS 的主价值不在单个页面，而在于项目上下文的连续性。用户从脚本到分镜，再到剪辑、配音、字幕、渲染与发布时，必须始终处于同一套壳层结构与导航心智中。

### 4.2 时间线是中枢，不是结果页

AI 剪辑工作台是产品中心。脚本、分镜、导入拆解、配音、字幕和资产中心都应把结果回流到时间线，而不是形成互相孤立的“生成页”。

### 4.3 AI 可见、可控、可回滚

AI 功能必须嵌入业务流程，比如脚本改写、分镜建议、AI 视频片段生成、TTS、字幕对齐、复盘建议。  
每次 AI 输出都必须有来源、版本、重试、覆盖确认和人工编辑入口。

### 4.4 先判断，再执行

对渲染、导出、发布、自动化任务、覆盖字幕、替换音轨、批量生成等操作，系统必须先展示对象范围、当前状态、风险项和恢复路径，再允许执行。

### 4.5 真实状态优先

UI 必须优先呈现真实数据、真实任务、真实健康状态和真实错误。  
禁止使用看起来像真实结果的假业务数据填满页面。

### 5.3 壳层区域定义

#### 5.3.1 Title Bar

职责：

1. 展示产品身份与当前项目上下文。
2. 提供全局搜索与最近访问入口。
3. 展示 AI Provider、Runtime、许可证和系统状态。
4. 提供系统设置和诊断入口。

交互要求：

1. 支持窗口拖动。
2. 支持全局搜索页面、项目、任务、资产和日志对象。
3. 状态区必须能快速打开系统设置或相关故障详情。

#### 5.3.2 Sidebar

职责：

1. 提供稳定的一级导航。
2. 用分组体现工作流阶段，而不是后台业务域。
3. 支持折叠态快速切换。

交互要求：

1. 展开宽度 280px。
2. 折叠宽度 64px。
3. 折叠态保留图标和 Tooltip。
4. 激活项使用品牌强调底和左侧强调条。
5. 分组标题在展开态显示，在折叠态隐藏。

#### 5.3.3 Content Host

职责：

1. 承载页面主内容区和页内工作流。
2. 容纳卡片、表格、表单、时间线、媒体预览和队列视图。
3. 作为页面滚动上下文，不让主导航承担页面级状态。

#### 5.3.4 Detail Panel

职责：

1. 展示选中对象的属性、日志、版本、预览和快捷动作。
2. 为时间线、资产、账号、设备、任务和发布对象提供次级工作区。

规范：

1. 默认宽度建议 360px，允许 320px 到 420px。
2. 可以关闭，但对核心工作台型页面应保持强关联。
3. 窗口较窄时可转为抽屉。

#### 5.3.5 Status Bar

职责：

1. 展示 Runtime、AI、任务、渲染、发布和同步状态。
2. 为后台任务提供持续反馈。
3. 在工作台型页面提供当前上下文状态，如选中轨、当前音色、字幕时间码或渲染进度。

1. 首启与许可证向导不在正常工作态的 Sidebar 中高频出现，仅在未完成激活或需要重配时显示。

## 8. 主题系统

### 8.1 主题目标

1. 统一支持 Light / Dark。
2. 不允许页面直接硬编码完整配色。
3. 所有页面通过语义令牌消费颜色和状态。
4. 创作与媒体类页面在 Dark 模式下也必须保持清晰层次与可读性。

### 8.2 推荐变量形式

```css
:root[data-theme="light"] {
  --surface-primary: #F5F8F8;
  --surface-secondary: #FFFFFF;
  --text-primary: #0F172A;
  --brand-primary: #00F2EA;
}

:root[data-theme="dark"] {
  --surface-primary: #0F2323;
  --surface-secondary: #0F172A;
  --text-primary: #F1F5F9;
  --brand-primary: #00F2EA;
}
```

### 8.3 主题切换规则

1. 切换主题不得打断当前项目编辑状态。
2. 时间线、波形、字幕轨、状态色在双主题下都必须可辨识。
3. 阴影、边框和下沉容器需要随主题同步调整，不能只换背景色。

## 9. 布局模板

### 9.1 Dashboard 模板

- 页面头部：标题、最近项目快捷入口、创建项目按钮。
- 主体：项目卡片、任务概览、状态摘要、异常提醒。
- Detail Panel：显示所选项目摘要和最近任务。

### 9.2 Studio 模板

适用于脚本、分镜、配音、字幕页：

- 左侧：列表或段落导航。
- 中部：主编辑区。
- 右侧：参数、版本、预览或日志。
- 底部 Status Bar：显示模型、当前版本、保存状态。

### 9.3 Timeline 模板

适用于 AI 剪辑工作台：

- 顶部：项目头、全局工具条、播放控制。
- 中部：预览窗口 + 时间线。
- 左侧：片段/轨道/镜头导航。
- 右侧：属性面板。
- 底部：时间码、渲染状态、当前选中轨信息。

### 9.4 Queue 模板

适用于渲染、自动化、发布：

- 顶部：筛选、批量操作、状态摘要。
- 主体：队列表格或任务列表。
- 右侧：任务详情与日志。

### 9.5 Settings 模板

- 左侧：设置分类。
- 中部：表单区域。
- 右侧：测试结果、健康状态、帮助信息。

## 10. 核心组件规范

### 10.1 Card

- 用于项目、资产、建议、日志摘要。
- 标准圆角 `radius.md`。
- Hover 可增强阴影，但不使用夸张位移动画。

### 10.2 Button

- 主按钮使用 `brand.primary`。
- 高风险动作使用 `status.error` 系列。
- 禁止用多个主按钮抢占注意力。

### 10.3 Input / Textarea / Prompt Editor

- 所有输入组件都必须支持错误反馈、禁用态和聚焦态。
- 大文本编辑器要支持草稿状态和自动保存提示。

### 10.4 Status Pill / Tag

- 用于表示任务状态、账号状态、设备健康、Provider 状态。
- 使用语义状态色，不用随机彩色标签。

### 10.5 Timeline Track

- 视频轨、音轨、字幕轨必须视觉上明显区分。
- 轨道头展示名称、静音/锁定/可见状态。
- 轨道内片段要支持选中、拖拽、冲突提示。

### 10.6 Detail Panel

- 默认承载属性、日志、版本、预览与快捷动作。
- 不作为唯一操作入口，但在工作台型页面必须是高价值辅助区域。

### 10.7 Empty State / Error State

- 空态强调下一步动作。
- 错态必须给出原因、日志入口和重试路径。

## 11. 页面设计规范

### 11.1 `setup_license_wizard`

- 类型：向导式页面。
- 重点：授权、目录、Runtime 健康和 AI 初始配置。

### 11.2 `creator_dashboard`

- 类型：工作台页。
- 重点：最近项目、待办、异常、运行健康。

### 11.3 `script_topic_center`

### 11.5 `ai_editing_workspace`

- 类型：核心工作台页。
- 重点：时间线、预览、AI 视频生成、片段替换、轨道联动。

### 11.6 `video_deconstruction_center`

### 11.7 `voice_studio`

- 类型：编辑页。
- 重点：音色、试听、语速、停顿、版本对比。
- Status Bar：当前模型、音色、生成状态。

### 11.8 `subtitle_alignment_center`

- 类型：编辑页。
- 重点：字幕生成、逐段编辑、时间码校正、模板预览。
- Detail Panel：样式、时间码和对齐诊断。

### 11.9 `asset_library`

- 类型：资产页。
- 重点：预览、筛选、标签、引用关系、版本。
- Detail Panel：资产详情与引用影响范围。

### 11.10 `account_management`

- 类型：管理页。

### 11.11 `device_workspace_management`

- 类型：管理页。

### 11.13 `publishing_center`

### 11.14 `render_export_center`

## 12. 响应式与桌面适配

### 12.1 断点建议

| 范围 | 设备 | 规则 |
| --- | --- | --- |
| `>= 1600px` | 宽屏工作站 | Sidebar 展开，Detail Panel 常驻，多列内容并排 |
| `1366px - 1599px` | 标准桌面 | Sidebar 展开，Detail Panel 按需展开 |
| `1200px - 1365px` | 紧凑桌面 | Sidebar 可折叠，部分工具条压缩 |
| `960px - 1199px` | 最小支持 | Detail Panel 优先转抽屉，局部操作并入更多菜单 |

### 12.2 响应规则

1. 优先折叠 Sidebar，再压缩页面工具条。
2. Detail Panel 优先转抽屉，不挤压时间线或主编辑区。
3. 时间线和预览区域在窄窗口中优先保留核心编辑空间。

## 13. 无障碍与可用性

### 13.1 可读性

- Light / Dark 都要保证足够对比度。
- 状态不能只靠颜色表达，必须同时有文案或图标。

### 13.2 键盘可达性

- Sidebar、表单、时间线关键操作、表格与任务列表必须可聚焦。
- 高频操作应预留快捷键空间。

### 13.3 反馈明确

- 保存、生成、渲染、发布、失败和重试必须有即时反馈。
- 自动保存必须让用户看见，不应悄悄覆盖。

## 14. 统一交付要求

1. 所有页面必须覆盖加载中、空状态、正常状态、错误状态。
2. 所有颜色、间距、圆角、阴影和状态色必须优先使用设计令牌，不写死。
3. 所有页面必须围绕真实 Runtime 数据、真实任务状态和真实日志设计。
4. AI 功能必须与当前页面上下文绑定，不做无来源的孤立弹窗能力。
5. 账号、设备、自动化、发布、渲染和复盘页面都必须与项目上下文可跳转。
6. 视觉语言要与 [docs/PRD.md](./PRD.md) 保持一致，不回到旧企业后台口径。

## 15. 结论

TK-OPS 的 UI 体系必须围绕“AI 视频创作中枢”展开。  
它的重点不是堆砌后台组件，而是通过稳定壳层、统一令牌、16 页结构、真实任务反馈和 AI 深接入工作流，把脚本、分镜、视频、声音、字幕、渲染、发布与自动化执行连接成一条连续的桌面创作链路。


## 16. V2 交互一致性要求
