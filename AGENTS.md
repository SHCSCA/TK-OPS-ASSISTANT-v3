# AGENTS.md - TK-OPS 代理协作手册

面向本仓库的 AI 编码代理与新加入开发者。  
目标：围绕 TK-OPS 当前的产品主线，持续交付可验证、可维护、真实数据驱动的桌面应用能力。

## 1) 产品北极星与语言要求

- 项目全局文案、注释、交互提示优先使用中文，必要技术名词可保留英文。
- 你在本仓库的默认目标：健壮性、全局一致性、真实交互、真实数据、样式一致。
- 禁止把“可点击原型”当最终结果：页面必须尽量由后端真实数据驱动。
- TK-OPS 当前不是团队协作后台，也不是经营系统，而是面向内容创作者的 `AI 视频创作中枢`。
- 产品第一目标平台是 TikTok，采用本地优先、一机一码授权、深度 AI 接入的桌面工作台模式。

## 1.1) 不可妥协的硬约束
- 编码采用 UTF-8 无 BOM，文档、注释、字符串必须保持中文可见，禁止出现乱码。
- 禁止大文件：不允许再出现单文件数千行实现，所有模块必须按职责边界合理拆分。
- 保证全局性：所有新开发必须遵循统一架构、统一规范、统一目录边界，避免局部特例破坏整体一致性。
- 全局采用中文注释：新增注释必须使用中文，并保持简洁，只解释代码意图、边界和复杂逻辑。
- 全局异常处理以及日志记录：所有异常必须被捕获、记录并转换为可见反馈；日志格式和错误处理方式必须统一。
- 全局配置总线：所有配置必须通过统一配置入口或配置总线管理，禁止页面、脚本、服务各自保存一套配置。
- 禁止伪业务范围回流：不得把订单、退款、商品、重 CRM、团队协作后台重新写回当前产品主线。
- 禁止高风险环境伪装：设备管理只允许管理真实 PC 工作区、浏览器实例和执行环境。
- UI 交互必须有异常、日志和状态反馈；禁止静默失败或只在控制台输出错误。
- 禁止在 UI 层直接拼接业务规则，必须通过服务层接口处理业务逻辑。
- 禁止在未核对 `docs/ARCHITECTURE-BOOTSTRAP.md` 的情况下擅自创建 `apps/desktop`、`apps/py-runtime` 目录或扩展路由、模块和模型。
- 禁止在未核对 `docs/PRD.md` 和 `docs/UI-DESIGN-PRD.md` 的情况下擅自扩页或改产品定位。
- 禁止新增“看起来像真实结果”的假业务数字；无真实后端数据时，必须使用中性空态或引导态。
- 禁止绕过授权开发后续功能；授权、一机一码、目录初始化、Runtime 健康检查必须走真实链路。
- 禁止账号、设备、工作区、浏览器实例与执行环境做假绑定；它们必须是真实对象。
- 禁止任何自动化、发布、渲染或 AI 长任务没有超时、失败提示和重试路径。
- 禁止在未核对 `docs/superpowers/plans/` 和 `docs/superpowers/specs/` 中相关计划和设计的情况下，直接进入实现阶段。
- UI 代码必须优先按页面、composable、helpers、types、styles 分层拆分；禁止把页面状态、服务调用和样式堆进单个超大文件。
- 所有数据请求必须统一通过 Runtime 适配层发起；禁止组件内直接 fetch。
- 样式必须优先使用设计令牌和 CSS Variables；同一改动需同时考虑 Light / Dark 双主题。
- Python 代码必须遵循统一规范，包括文件头、导入顺序、类型标注、命名、日志、异常记录、事务控制、错误处理、配置管理和改动原则。
- UI 交互必须至少覆盖加载中、空状态、正常状态、错误状态；必须处理异常、日志、重试、取消和状态反馈；必须同步处理缓存失效、任务刷新与全局状态更新；必须补齐至少一条改动相关测试。
- UI设计要求至少覆盖桌面宽屏和紧凑窗口两种布局，特别是时间线、任务队列、状态栏、Detail Panel等核心交互组件。
- UI设计必须与当前产品定位一致，禁止把产品漂回旧后台口径、假数据演示、旧壳路径或失控的自动化范围。
- UI要有设计感，禁止直接把功能堆在一起，必须有合理的视觉层次和交互引导。
- UI必须有明确的状态反馈，禁止让用户无感知地等待或不确定操作结果。
- UI要有一致的风格和交互模式，禁止同一页面或功能里出现多种不同的设计语言或交互方式。
- UI必须适配不同的屏幕尺寸和使用场景，禁止只针对某一种设备或窗口大小设计。
- UI交互要有极高体验感,要好看、流畅、易用,禁止出现卡顿、闪烁、错位等问题。
- UI设计以及交互可以参考 https://github.com/DavidHDev/react-bits.git以及https://www.unicorn.studio/
- 后端开发完成要给到明确的接口文档给到前端，前端开发完成要给到明确的接口调用文档给到后端，禁止接口不明确导致的前后端对接问题。
- 接口文档以及调用文档必须包含接口地址、请求方法、请求参数、返回结果、错误码和示例，禁止模糊不清的接口说明。
- 前后端接口必须保持一致，禁止接口变更导致的前后端不兼容问题。
- 前后端接口必须有版本控制，禁止接口变更导致的版本冲突问题。
- 前后端接口文档唯一且及时更新以及追加，禁止接口文档过时导致的对接问题。
- 前后端接口必须有明确的错误处理机制，禁止接口调用失败导致的用户体验问题。


## 2) 文档真源与冲突处理

- 产品真源：`docs/PRD.md`
- UI 真源：`docs/UI-DESIGN-PRD.md`
- 实现骨架真源：`docs/ARCHITECTURE-BOOTSTRAP.md`
- 协作与工程约束真源：`AGENTS.md`

处理原则：

- 当产品范围、页面树、模型边界冲突时，以 `docs/PRD.md` 为准。
- 当视觉语言、壳层结构、页面布局冲突时，以 `docs/UI-DESIGN-PRD.md` 为准。
- 当目录树、路由归属、模块责任和模型落点冲突时，以 `docs/ARCHITECTURE-BOOTSTRAP.md` 为准。
- 当实现方式、工程约束冲突时，以 `AGENTS.md` 为准。
- 若三者出现明显矛盾，不要擅自猜测，应先更新文档，再进入实现。

## 3) 当前项目阶段与架构目标

当前仓库仍处于“文档真源已重写、实现骨架待逐步落地”的阶段。  
因此：

- 不要假设所有目标目录、脚本和入口已经存在。
- 如果某个目标路径还不存在，应在已批准的计划下创建到目标边界内，而不是回退到旧壳目录。
- 若命令、脚本、目录或构建文件尚未落地，先验证存在性，再引用它们。

目标架构主线：

- 桌面壳：`Tauri 2 + Vue 3 + TypeScript + Vite`
- 业务运行时：`Python + FastAPI + SQLAlchemy + Alembic`
- 前端状态与路由：`Pinia + Vue Router`
- 本地数据：`SQLite + 本地项目工作区`
- 通信：`HTTP + WebSocket`
- 媒体流水线：独立渲染/转码链路，按 FFmpeg 类能力规划
- AI 体系：多 Provider 适配层，覆盖脚本、视频、TTS、字幕和资产分析

## 4) 目标目录边界

新功能默认只允许落在以下边界中：

- `apps/desktop`
- `apps/py-runtime`
- `tests`
- `docs`

旧壳目录处理原则：

- `desktop_app/` 仅允许作为迁移参考、旧模型参考或历史实现参考。
- 旧壳不得重新进入运行、联调、打包或发布链路。

目标路径约定：

- 后续新目录、路由、模块和模型的落点以 `docs/ARCHITECTURE-BOOTSTRAP.md` 为准。
- 前端壳层建议放在 `apps/desktop/src/layouts/`
- 前端页面建议放在 `apps/desktop/src/pages/`
- 前端模块建议放在 `apps/desktop/src/modules/`
- 路由真源建议放在 `apps/desktop/src/app/router/`
- Runtime 入口建议放在 `apps/py-runtime/src/`
- 测试统一放在 `tests/`

若上述路径尚未存在：

- 在已批准计划下创建它们。
- 不要因为缺目录就把新实现塞回旧目录或单文件脚本。

## 5) 当前产品范围与页面树

当前正式产品范围固定为 16 页：

1. 首启与许可证向导
2. 创作总览
3. 脚本与选题中心
4. 分镜规划中心
5. AI 剪辑工作台
6. 视频拆解中心
7. 配音中心
8. 字幕对齐中心
9. 资产中心
10. 账号管理
11. 设备与工作区管理
12. 自动化执行中心
13. 发布中心
14. 渲染与导出中心
15. 复盘与优化中心
16. AI 与系统设置

写代码时要遵守以下边界：

- 账号管理只服务发布与执行闭环，不扩展成经营后台。
- 设备管理只服务真实工作区、浏览器实例和执行环境，不实现高风险能力。
- AI 剪辑工作台是产品中枢，相关能力应优先向它回流，而不是形成孤立工具页。
- 自动化、渲染、发布和复盘从属于创作中枢，不单独膨胀成另一套后台。

## 6) 核心模型与接口约束

### 6.1 项目主模型

固定主链：

`Project -> Script -> Storyboard -> Timeline -> VoiceTrack -> SubtitleTrack -> RenderTask`

要求：

- 从零创作与导入拆解必须最终回到同一 Project。
- 不允许为不同入口创建两套彼此割裂的数据模型。

### 6.2 账号与设备模型

固定主链：

`Account -> DeviceWorkspace -> ExecutionBinding`

要求：

- 发布、自动化与诊断必须能回溯到明确的账号和工作区。
- 工作区必须是真实存在、可审计、可恢复的本地对象。

### 6.3 AI 能力接口

V1 统一按能力接口分层：

- `ScriptProvider`
- `VideoGenerationProvider`
- `TTSProvider`
- `SubtitleAlignmentProvider`
- `AssetAnalysisProvider`

要求：

- 页面不直接硬编码供应商实现。
- AI 调用必须有日志、耗时、失败原因和版本记录。

### 6.4 Runtime 契约

前后端统一使用 Runtime HTTP / WebSocket 通信。  
Runtime 返回统一 JSON 信封：

- 成功：`{ "ok": true, "data": ... }`
- 失败：`{ "ok": false, "error": "..." }`

要求：

- 错误信息应可直接转为中文 UI 提示。
- 数据变化后必须同步刷新缓存、队列和状态广播。

## 7) 页面实现检查清单

新增或改造页面时，至少检查以下内容：

1. 是否属于当前 16 页之一，或是否已先更新产品文档。
2. 是否在正确目录边界下新增页面、模块、样式和测试。
3. 是否接入真实 Runtime 数据，而不是停留在假数据。
4. 是否补齐加载中、空状态、正常状态、错误状态。
5. 是否处理异常、日志、重试、取消和状态反馈。
6. 是否接入统一配置总线，而不是本地散落保存配置。
7. 是否同步处理缓存失效、任务刷新与全局状态更新。
8. 是否补齐至少一条改动相关测试。

## 8) Python 代码规范

- 文件头：`from __future__ import annotations`
- 导入顺序：标准库 -> 第三方库 -> 本地模块
- 类型：优先完整类型标注，使用 `|` 联合类型
- 命名：`snake_case`（函数/变量）、`PascalCase`（类）、`UPPER_SNAKE_CASE`（常量）
- 日志：模块级 `log = logging.getLogger(__name__)`
- 异常记录：使用 `log.exception(...)`
- 事务：显式控制事务边界，失败必须可回滚
- 错误处理：
  - UI 可感知错误必须转为明确中文错误信息
  - 不得把 traceback 直接暴露给用户
- 配置：统一走配置入口或配置总线
- 改动原则：小步、最小必要修改，避免无关重构

## 9) Vue / TypeScript / CSS 规范

- 页面代码优先按页面、composable、helpers、types、styles 分层拆分。
- 禁止把页面状态、服务调用和样式堆进单个超大文件。
- 所有数据请求统一通过 Runtime 适配层发起。
- 交互异常必须有 UI 反馈，不得静默失败。
- 样式必须优先使用设计令牌和 CSS Variables。
- 同一改动需同时考虑 Light / Dark 双主题。
- 对时间线、任务队列、状态栏、Detail Panel 的修改，必须考虑桌面宽屏和紧凑窗口两种布局。

## 10) 数据真实性、授权与设备边界

- 禁止新增“看起来像真实结果”的假业务数字。
- 无真实后端数据时，使用中性空态或引导态。
- 授权、一机一码、目录初始化、Runtime 健康检查必须走真实链路。
- 账号、设备、工作区、浏览器实例与执行环境必须是真实对象，不做假绑定。
- 任何自动化、发布、渲染或 AI 长任务都必须有超时、失败提示和重试路径。

## 11) 测试与验证

- 功能改动至少跑“改动相关测试 + 1 条主链路回归”。
- 文档更新后至少做一次文本级一致性检查，确认范围、页树和术语未冲突。
- 前端改动：至少覆盖页面状态、数据契约或关键交互。
- Runtime 改动：至少覆盖服务层、实体层或接口契约。
- 在不存在测试、脚本或构建文件的情况下，不要伪称已验证通过；先说明缺失事实，再给出已完成的检查。

## 12) 常见陷阱

- 不要把旧 PySide6 壳拉回运行链路。
- 不要把产品范围漂回订单、退款、商品或团队后台。
- 不要绕过服务层直接在 UI 层拼接业务规则。
- 不要把配置写死在页面、脚本或服务里。
- 不要在 UI 文案里掺入大量英文占位和开发者语气。
- 不要跳过异常、日志和状态反馈。
- 不要在未核对 `docs/ARCHITECTURE-BOOTSTRAP.md` 的情况下擅自创建 `apps/desktop`、`apps/py-runtime` 目录或扩展路由、模块和模型。
- 不要在未核对 `docs/PRD.md` 和 `docs/UI-DESIGN-PRD.md` 的情况下擅自扩页或改产品定位。

## 13) 交付标准

- 代码或文档遵循统一产品主线与 16 页结构。
- 无单文件数千行实现，模块拆分清晰。
- 新链路中不存在双壳并行运行或旧壳回流。
- 页面有真实数据路径或明确的空态/引导态。
- 异常路径可见、可恢复、可追踪。
- 配置、日志、任务和状态流转遵循统一系统能力。
- 文案、交互、样式与当前产品定位一致。

## 14) Superpowers 工作流规则

- 当前仓库内，所有非简单工程任务强制先走 superpowers 工作流。
- 固定顺序：先生成 `docs/superpowers/plans/YYYY-MM-DD-topic.md`，经审批通过后，再生成 `docs/superpowers/specs/YYYY-MM-DD-topic-design.md`，最后才进入实现。
- 非简单工程任务默认包括：多文件改动、架构调整、前后端契约变更、模块拆分、主链路功能改造、测试体系重构。
- 可不强制走完整流程的范围仅限：纯问答、只读分析、单文件低风险小修、纯文案微调。
- 所有大改必须分阶段推进。plan 中必须明确阶段目标、文件地图、验证方式、边界与回退点；未经批准不得跳过 plan 直接写 spec 或直接编码。
- 当实现范围明显超出已批准的 plan / spec 时，必须先更新文档，再继续开发。

## 15) 总结

在 TK-OPS 仓库里，代理的默认职责不是“把任何需求快速做成页面”，而是围绕 `AI 视频创作中枢` 这条主线，持续交付真实、可验证、可维护的桌面产品能力。  
如果某项改动会让产品漂回旧后台口径、假数据演示、旧壳路径或失控的自动化范围，那么它就是错误方向，应先停下并回到文档对齐。
